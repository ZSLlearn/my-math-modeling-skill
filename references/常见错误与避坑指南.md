# 数学建模竞赛常见错误与避坑指南

本文档总结了数学建模竞赛中参赛队伍最常犯的错误，按角色分类整理，帮助团队在比赛过程中避免低级失误，提高论文质量。

---

## 一、建模分析阶段（建模手）

### ❌ 错误 1：模型选择过于简单

| 表现                               | 后果                 | 正确做法                                    |
| ---------------------------------- | -------------------- | ------------------------------------------- |
| 只使用最简单的线性回归解决预测问题 | 评委认为建模能力不足 | 至少使用2-3种模型对比，或对基础模型进行改进 |
| 单一方法解决所有子问题             | 缺乏层次感和深度     | 针对不同子问题的特点选择不同方法            |

### ❌ 错误 2：盲目追求复杂模型

| 表现                       | 后果                         | 正确做法                                             |
| -------------------------- | ---------------------------- | ---------------------------------------------------- |
| 不管数据量大小就上深度学习 | 过拟合、解释困难、被评委质疑 | 优先选择"简单有效"的模型，复杂模型仅在确实需要时使用 |
| 堆砌多种高级算法           | 逻辑混乱，无法合理解释       | 每个模型的使用都要有明确的理由                       |

### ❌ 错误 3：忽略数据预处理

| 表现                 | 后果                               | 正确做法                                             |
| -------------------- | ---------------------------------- | ---------------------------------------------------- |
| 直接使用原始数据建模 | 异常值影响结果、量纲不一致导致错误 | 先进行缺失值处理、异常值检测、标准化/归一化          |
| 不检查数据质量       | 模型结果可能完全错误               | 使用 `df.describe()`、`df.info()` 等查看数据概况 |

### ❌ 错误 4：符号定义混乱

| 表现                         | 后果         | 正确做法                   |
| ---------------------------- | ------------ | -------------------------- |
| 同一变量在不同位置用不同符号 | 评委阅读困难 | 建立术语表，全文统一符号   |
| 符号定义不清晰               | 公式无法理解 | 每个符号首次出现时必须定义 |

### ❌ 错误 5：不考虑模型的可行性

| 表现                   | 后果               | 正确做法                     |
| ---------------------- | ------------------ | ---------------------------- |
| 选择的模型计算量过大   | 比赛时间内无法求解 | 评估计算复杂度，准备简化方案 |
| 依赖无法获取的外部数据 | 模型无法实现       | 建模前确认数据可用性         |

## 二、代码实现阶段（编程手）

### ❌ 错误 6：代码不运行就直接提交思路

| 表现                 | 后果                  | 正确做法                             |
| -------------------- | --------------------- | ------------------------------------ |
| 只写代码不运行       | 代码有bug，结果不存在 | 每段代码必须实际运行并验证输出       |
| 假设代码"应该没问题" | 论文中的数据是编造的  | 运行 → 检查报错 → 修正 → 重新运行 |

### ❌ 错误 7：创建多个简化版代码文件

| 表现                                        | 后果                   | 正确做法                            |
| ------------------------------------------- | ---------------------- | ----------------------------------- |
| 创建 `问题一_简化版.py`、`问题一_v2.py` | 代码混乱，团队无法协作 | 始终在原文件中修改，用 Git 管理版本 |
| 遇到困难就换简单方法                        | 结果不符合题目要求     | 分析错误原因，在原文件修复          |

### ❌ 错误 8：图表质量差

| 表现                     | 后果                 | 正确做法                          |
| ------------------------ | -------------------- | --------------------------------- |
| 图表没有标题、坐标轴标签 | 评委看不懂           | 每张图必须有清晰的标题和轴标签    |
| 直接使用默认配色         | 不美观、不专业       | 使用 `seaborn` 风格或自定义配色 |
| 图片分辨率太低           | 打印后模糊           | 使用 `dpi=300` 以上保存图片     |
| 中文字体乱码             | 图上的中文显示为方块 | 提前配置中文字体（SimHei 等）     |

### ❌ 错误 9：结果只输出不分析

| 表现                 | 后果               | 正确做法               |
| -------------------- | ------------------ | ---------------------- |
| 只输出数值，不做解释 | 评委不知道结果含义 | 每个结果都要有文字分析 |
| 不检查结果的合理性   | 可能提交荒谬的答案 | 用常识判断结果是否合理 |

### ❌ 错误 10：不保存中间结果

| 表现                 | 后果                     | 正确做法                             |
| -------------------- | ------------------------ | ------------------------------------ |
| 每次运行都从头计算   | 浪费时间，比赛后期来不及 | 关键计算结果保存到文件（.npy、.csv） |
| 不确定哪个结果是对的 | 重复劳动                 | 使用版本化的文件名记录不同参数的结果 |

### ❌ 错误 11：代码没有注释

| 表现       | 后果                           | 正确做法                             |
| ---------- | ------------------------------ | ------------------------------------ |
| 代码零注释 | 论文手看不懂，评委看附录看不懂 | 关键步骤添加注释，函数添加 docstring |

## 三、论文撰写阶段（论文手）

### ❌ 错误 12：摘要写得不好

> **这是最致命的错误！** 摘要是评委第一眼看到的内容，直接决定第一印象。

| 表现                 | 后果                   | 正确做法                                                        |
| -------------------- | ---------------------- | --------------------------------------------------------------- |
| 摘要太长（超过1页）  | 违反竞赛规定           | 严格控制在1页以内                                               |
| 摘要太短（不到半页） | 信息量不足             | 摘要应包含：问题→方法→结果→结论 的完整逻辑链                 |
| 摘要只讲方法不写结果 | 评委不知道你得出了什么 | 必须在摘要中给出具体数值结果                                    |
| 摘要只是章节标题罗列 | 像目录不像摘要         | 用连贯的段落自然陈述，避免"首先...其次...然后...最后"的僵硬模板 |

### ❌ 错误 13：缺少灵敏度分析

> **这是最常见的扣分点！**

| 表现               | 后果                       | 正确做法                                 |
| ------------------ | -------------------------- | ---------------------------------------- |
| 完全不做灵敏度分析 | 严重扣分                   | 至少选择1-2个关键参数进行灵敏度分析      |
| 做了但不分析结果   | 做了等于没做               | 必须对灵敏度曲线进行文字解读             |
| 随意选参数分析     | 对不重要的参数分析没有意义 | 选择核心参数（权重、阈值、关键系数）分析 |

### ❌ 错误 14：缺少模型检验

| 表现                 | 后果               | 正确做法                         |
| -------------------- | ------------------ | -------------------------------- |
| 只给出结果不说准不准 | 评委质疑模型有效性 | 必须给出误差指标（RMSE、R² 等） |
| 不给检验方法的过程   | 看不出科学性       | 展示检验方法的具体计算过程       |
| 没有多模型对比       | 看不出模型优劣     | 至少与1-2个基准模型对比          |

### ❌ 错误 15：图表引用不规范

| 表现           | 后果               | 正确做法                        |
| -------------- | ------------------ | ------------------------------- |
| 图没有编号     | 无法引用           | 每张图标注"图X"                 |
| 文中不引用图表 | 图和文分离         | 文中必须有"如图X所示"等引用语句 |
| 图表放在附录   | 评委看不到核心结果 | 重要图表必须在正文中展示        |

### ❌ 错误 16：AI痕迹过重

| 表现                                 | 后果                 | 正确做法                   |
| ------------------------------------ | -------------------- | -------------------------- |
| 大量使用"首先...其次...然后...最后"  | 一眼看出是AI写的     | 使用自然段落，适当变化句式 |
| "具有重要的理论意义和实践价值"等套话 | 空洞无物             | 用具体内容代替套话         |
| 每段开头格式高度一致                 | 缺乏人类写作的变化性 | 交替使用不同句式开头       |

### ❌ 错误 17：论文格式不符合要求

| 表现             | 后果         | 正确做法                             |
| ---------------- | ------------ | ------------------------------------ |
| 字体、字号不统一 | 格式分扣完   | 严格按竞赛手册设置格式               |
| 页码缺失或错位   | 影响评审体验 | 检查页码连续性                       |
| 参考文献格式混乱 | 学术规范扣分 | 统一使用一种引用格式（GB/T 7714 等） |

## 四、团队协作阶段

### ❌ 错误 18：三个角色完全串行

| 表现                       | 后果         | 正确做法                                   |
| -------------------------- | ------------ | ------------------------------------------ |
| 等建模手完全做完再开始编程 | 浪费大量时间 | 建模手分析第一问时，编程手就可以搭代码框架 |
| 等所有代码跑完再写论文     | 最后来不及   | 论文手先写问题重述、模型假设等前置章节     |

### ❌ 错误 19：不备份文件

| 表现             | 后果           | 正确做法                 |
| ---------------- | -------------- | ------------------------ |
| 只有一个本地副本 | 电脑出问题全丢 | 使用 Git 或网盘实时同步  |
| 最后一天才整合   | 版本混乱       | 每完成一个阶段就整合一次 |

### ❌ 错误 20：各自为战缺乏沟通

| 表现                             | 后果               | 正确做法             |
| -------------------------------- | ------------------ | -------------------- |
| 建模手和编程手对算法的理解不一致 | 代码实现与模型不符 | 编程前先对齐算法流程 |
| 论文手写的与实际情况不符         | 论文与结果脱节     | 论文手随时同步进展   |

## 五、终极检查清单

提交论文前，逐项确认：

### 内容检查

- [ ] 摘要是否在1页以内、包含完整的问题-方法-结果-结论？
- [ ] 是否包含**灵敏度分析**章节？
- [ ] 是否包含**模型检验**章节（含RMSE/R²等指标）？
- [ ] 是否包含**模型优缺点与改进**章节？
- [ ] 每张图表是否有编号和标题？
- [ ] 正文中是否引用了所有图表？
- [ ] 所有符号是否在符号说明表中定义？
- [ ] 参考文献是否格式统一、数量不少于5篇？

### 结果检查

- [ ] 所有子问题是否都给出了答案？
- [ ] 结果数值是否有量纲/单位？
- [ ] 结果是否合理（用常识判断）？
- [ ] 结果是否与代码输出一致（不是手填的）？

## 七、历年典型翻车案例

| 案例                                         | 问题                       | 教训                     |
| -------------------------------------------- | -------------------------- | ------------------------ |
| 某队选了深度学习模型                         | 数据量只有50条，严重过拟合 | 模型复杂度要与数据量匹配 |
| 某队摘要写了3页                              | 违反规定被扣分             | 摘要必须≤1页            |
| 某队所有图都是默认matplotlib风格             | 整体印象差                 | 花时间美化图表是值得的   |
| 某队没做灵敏度分析                           | 直接降档                   | 灵敏度分析是必做项       |
| 某队最后一个小时还在改代码                   | 论文来不及检查             | 遵守时间节点红线         |
| 某队代码全是bug没运行过                      | 论文中的数据是自己编的     | 代码必须实际运行         |
| 某队三个子问题用了三个完全不同毫无关联的模型 | 论文缺乏整体性             | 模型之间应有逻辑联系     |
| 某队论文中符号前后不一致                     | 评委看不懂                 | 建立严格的术语表         |

---

**切记**：数学建模竞赛考察的是**综合能力**——分析问题、建立模型、编写代码、撰写论文，任何一个环节的短板都会影响最终成绩。这份指南不是为了吓唬你，而是希望你能避开前人踩过的坑，把有限的时间用在有价值的地方。
